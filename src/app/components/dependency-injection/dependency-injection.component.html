Dependency Injection (DI) to wzorzec projektowy stosowany w programowaniu, w tym w Angularze, który ma na celu zapewnienie elastyczności, łatwości testowania i zwiększenia modularności aplikacji. W DI obiekty nie tworzone są bezpośrednio przez inne obiekty, ale są im dostarczane przez zewnętrzne "zależności".

W Angularze DI jest wbudowany i jest używany do zarządzania zależnościami między komponentami, usługami, pipeami itp. Dzięki DI możemy wstrzykiwać zależności w konstruktorach lub w innych metodach obiektów, a framework Angular automatycznie dostarczy odpowiednie instancje tych zależności.

Przykład użycia DI w Angularze może wyglądać tak:

typescript
import {{'{'}} Component, Injectable } from '@angular/core';

        @Injectable()
        class MyService {{'{'}}
          // Implementacja serwisu
        }
        
        @Component({{'{'}}
          selector: 'my-component',
          template: `<h1>Hello Angular!</h1>`,
          providers: [MyService] // Rejestracja usługi w kontekście komponentu
        })
        class MyComponent {{'{'}}
          constructor(private myService: MyService) {{'{'}}
            // Wstrzyknięcie usługi MyService przez konstruktor
          }
        }

W powyższym przykładzie MyService jest klasą usługi, która zostaje zarejestrowana jako dostępna do wstrzykiwania. Następnie w klasie MyComponent korzystamy z DI, aby wstrzyknąć instancję MyService poprzez konstruktor.

Angular jest odpowiedzialny za tworzenie i dostarczanie odpowiednich instancji zależności. Możemy również skonfigurować DI na różne sposoby, takie jak dostarczanie jednej instancji usługi dla całej aplikacji (singleton) lub dostarczanie oddzielnych instancji dla różnych komponentów.

DI w Angularze ma wiele zalet, w tym:

  • Łatwość testowania: Możemy łatwo dostarczyć fałszywe (mock) zależności podczas testowania jednostkowego.
  • Elastyczność: Możemy łatwo wymieniać zależności lub dostarczać różne implementacje zależności bez modyfikowania kodu komponentów.
  • Reużywalność: Możemy używać tych samych usług w różnych częściach aplikacji, co prowadzi do większej modularności i łatwiejszego ponownego wykorzystania kodu.

Ważne jest również zrozumienie, że DI w Angularze opiera się na hierarchicznym drzewie komponentów. Jeśli zależność nie jest dostarczona bezpośrednio przez komponent, Angular będzie szukał jej w hierarchii nadrzędnych komponentów aż do najwyższego poziomu aplikacji.
